## 什么是负载均衡？

**负载均衡器**将请求分发到多台服务器，防止单台服务器过载。

```
                       ┌─ 服务器 A
客户端 → 负载均衡器 ─┼─ 服务器 B
                       └─ 服务器 C
```

没有负载均衡，所有流量都打到一台机器——即使加了更多服务器也没用。

## 负载均衡算法

### 轮询（Round Robin）
按顺序分发请求：A → B → C → A → B → C…

简单均匀，但不考虑服务器当前负载和请求复杂度。

### 最少连接（Least Connections）
路由到当前活跃连接数最少的服务器。

适合请求处理时间差异大的场景（有些请求 1ms，有些 500ms）。

### IP 哈希（IP Hashing）
对客户端 IP 做哈希，确保同一客户端始终路由到同一台服务器。

适用于需要会话亲和性的场景。

### 加权轮询（Weighted Round Robin）
给服务器分配权重，权重为 3 的服务器收到的请求是权重为 1 的 3 倍。

适合服务器配置不同的场景。

## L4 vs L7 负载均衡

| | L4（传输层）| L7（应用层）|
|--|------------|------------|
| **工作层级** | TCP/UDP 层 | HTTP 层 |
| **路由依据** | IP + 端口 | URL、Header、Cookie |
| **速度** | 更快 | 较慢 |
| **智能路由** | 不支持 | 支持 |
| **典型产品** | AWS NLB | AWS ALB、Nginx |

**L4**：无脑转发数据包——快但不智能。

**L7**：读取请求内容——可以将 `/api/*` 路由到一个集群，`/static/*` 路由到另一个，或基于 `Authorization` Header 路由。

## 健康检查

负载均衡器持续探测服务器是否存活，健康检查失败的服务器会被从池中移除。

```
负载均衡器 → GET /health → 服务器 A（200 OK）✓
           → GET /health → 服务器 B（超时）✗ 移除
           → GET /health → 服务器 C（200 OK）✓
```

## 粘性会话（Sticky Sessions）

有些应用把会话状态存在服务器本地。如果每次请求打到不同服务器，用户会丢失会话。

**粘性会话**（会话亲和性）确保同一用户始终路由到同一台服务器。

更好的方案：把会话存到 Redis，任意服务器都能处理任意用户的请求。

## 负载均衡器的高可用

负载均衡器本身可能成为单点故障，解决方案：
- **主备模式**：一个负载均衡器主用，另一个备用
- **主主模式**：多个负载均衡器共同承载流量（DNS 轮询）
- **托管服务**：AWS ALB、GCP Load Balancer 自动处理高可用

## 关键问题

> _Q: 什么是负载均衡器？为什么需要它？_

负载均衡器将请求分发到多台服务器，防止过载，提高可用性，支持横向扩展。没有它，加再多服务器也没用——所有流量还是打到一台机器。它还提升了容错能力：某台服务器宕机时，负载均衡器停止向它转发请求。

> _Q: L4 和 L7 负载均衡的区别是什么？_

L4 在传输层工作——基于 IP 和端口转发 TCP/UDP 数据包，不检查内容。L7 在应用层工作——可以检查 HTTP Header、URL 和 Cookie，实现更智能的路由，比如将 API 流量和静态资源分发到不同集群。

> _Q: 什么是粘性会话？有什么问题？_

粘性会话通过 IP 哈希或 Session Cookie 将用户固定到同一台服务器。适用于会话状态存在服务器本地的场景。问题是它降低了负载均衡效果——某台服务器慢，它的"粘性"用户就只能等。更好的方案是把会话存到外部存储（如 Redis）。

> _Q: 轮询和最少连接的区别是什么？_

轮询按顺序均匀分发请求，不考虑服务器当前负载。最少连接将请求路由到活跃连接数最少的服务器，对处理时间差异大的请求更自适应。负载均匀时用轮询；请求耗时差异明显时用最少连接。
