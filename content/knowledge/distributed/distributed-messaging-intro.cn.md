## 什么是消息队列？

**消息队列**是一个让服务之间通过共享缓冲区**异步通信**的组件。发送方（生产者）把消息放进队列就可以继续干别的了——不需要等接收方（消费者）处理完。

可以把它想象成发邮件：你发完就继续工作，收件人什么时候读是他的事。

## 为什么要用消息队列？

没有消息队列时，服务之间直接调用（同步）：

```
订单服务 → 支付服务 → 库存服务 → 通知服务
```

如果支付服务变慢或宕机，整条链路都会卡住。有了消息队列：

```
订单服务 → [队列] ← 支付服务
                  ← 库存服务
                  ← 通知服务
```

好处：
- **解耦** — 服务之间不需要互相感知
- **韧性** — 消费者宕机时，消息在队列里等着
- **削峰填谷** — 平滑流量波动，消费者按自己的节奏处理
- **可扩展** — 增加更多消费者来提升处理速度

## 两种核心模式

### 点对点（Queue）
一个生产者，一个消费者。每条消息**只被处理一次**。

```
生产者 → [队列] → 消费者 A
```

适用场景：任务处理、作业队列（比如：压缩这张图片）

### 发布/订阅（Pub/Sub）
一个生产者，**多个消费者**。每个消费者都能收到每条消息的副本。

```
生产者 → [Topic] → 消费者 A
                 → 消费者 B
                 → 消费者 C
```

适用场景：事件广播（比如："订单已下单" → 同时通知支付、库存、邮件服务）

## 常见消息队列系统

| 系统 | 适合场景 |
|------|---------|
| **Kafka** | 高吞吐事件流、日志聚合 |
| **RabbitMQ** | 复杂路由、任务队列、低延迟 |
| **AWS SQS** | AWS 上的简单托管队列 |
| **Redis Streams** | 用 Redis 实现的轻量级队列 |

## 关键问题

> _Q: 同步通信和异步通信有什么区别？_

同步通信是调用方等待响应后才继续执行（比如 HTTP 请求）。异步通信是调用方发完消息就继续，不等待接收方处理。消息队列让服务之间实现异步通信。

> _Q: 队列（Queue）和主题（Topic/Pub/Sub）有什么区别？_

队列把每条消息只投递给一个消费者（点对点）。主题把每条消息投递给所有订阅者（发布/订阅）。任务分发用队列，需要多个服务响应同一事件时用主题。

> _Q: 消息队列如何提升系统韧性？_

如果下游服务宕机，消息会堆积在队列里而不是直接丢失。服务恢复后会处理积压。没有队列的话，上游服务要么失败，要么需要自己实现重试逻辑——队列优雅地吸收了这个故障。

> _Q: 什么是背压（Backpressure）？_

背压是指消费者处理速度跟不上消息生产速度，导致队列持续增长。解决方案包括横向扩展消费者、对生产者限流，或设置队列大小上限并处理溢出情况。
