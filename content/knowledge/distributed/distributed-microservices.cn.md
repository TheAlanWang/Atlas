## 什么是单体架构？

**单体架构（Monolith）** 是一个单一可部署单元，所有功能都在同一个代码库里——用户管理、支付、订单、通知全部打包在一起。

```
┌─────────────────────────────────┐
│           单体应用               │
│  用户 │ 订单 │ 支付 │ 通知 │ …  │
│             共享数据库            │
└─────────────────────────────────┘
```

早期开发简单，但随着代码库增长会越来越难维护。

## 什么是微服务？

**微服务**将应用拆分成小型独立服务——每个服务有自己的代码库、数据库和部署流水线。

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│  用户    │  │  订单    │  │  支付    │
│  服务    │  │  服务    │  │  服务    │
│  数据库  │  │  数据库  │  │  数据库  │
└──────────┘  └──────────┘  └──────────┘
      ↕              ↕             ↕
           API / 消息队列
```

每个服务：
- 拥有自己的数据
- 独立部署
- 通过 API 或消息队列通信
- 可以用不同语言编写

## 单体 vs 微服务对比

| | 单体 | 微服务 |
|--|------|--------|
| **部署** | 整体一起部署 | 每个服务独立部署 |
| **扩展** | 只能整体扩展 | 按需扩展特定服务 |
| **开发** | 早期简单 | 复杂但可并行 |
| **故障** | 一个 bug 可能拖垮整体 | 故障隔离 |
| **数据** | 共享数据库 | 每个服务独立数据库 |
| **延迟** | 低（进程内调用）| 较高（网络调用）|
| **适合** | 小团队、早期阶段 | 大团队、高并发 |

## 怎么选？

**从单体开始，当：**
- 团队规模小（< 10 人）
- 产品还在找 PMF
- 还不清楚服务边界在哪里

**迁移到微服务，当：**
- 不同模块需要独立扩展
- 多个团队需要独立部署
- 某个服务成为瓶颈

> "不要一开始就用微服务。先用单体，感受到痛点时再拆分。" — Martin Fowler

## 常见微服务模式

### API 网关
所有客户端请求的统一入口。负责请求路由、鉴权、限流和日志，转发给对应服务。

```
客户端 → API 网关 → 用户服务
                 → 订单服务
                 → 支付服务
```

### 服务发现
服务启动时向注册中心（如 Consul、Kubernetes DNS）注册自己。其他服务动态查找地址，不需要硬编码 IP。

### 熔断器（Circuit Breaker）
如果下游服务持续失败，暂时停止调用，防止故障级联扩散。

```
正常：  A → B（成功）
失败中：A → B（失败）→ A → B（失败）→ 熔断器打开
打开：  A → 降级响应（30秒内不调用 B）
半开：  尝试调用 B → 成功则关闭熔断器
```

## 关键问题

> _Q: 单体架构和微服务架构的主要区别是什么？_

单体是单一可部署单元，共享代码库和数据库。微服务将应用拆分为独立服务，每个服务有自己的代码库、数据库和部署生命周期。微服务支持独立扩展和部署，但引入了网络复杂性和分布式系统挑战。

> _Q: 微服务的缺点是什么？_

微服务引入了运维复杂性：分布式链路追踪、服务间网络延迟、跨数据库数据一致性、服务发现，以及更复杂的部署基础设施。本地开发也更困难，且故障面更大。

> _Q: 什么是 API 网关？有什么作用？_

API 网关是所有微服务前面的统一入口，处理鉴权、限流、请求路由、日志等横切关注点，让各服务不需要各自实现这些功能。同时简化了客户端接口——客户端只需对接一个端点，而不是几十个。

> _Q: 什么是熔断器模式？_

熔断器监控对下游服务的调用。失败次数超过阈值时，"打开"熔断器——在一段时间内停止调用该服务并返回降级响应。这防止了故障级联：一个慢服务导致所有上游服务堆积请求并一起崩溃。
