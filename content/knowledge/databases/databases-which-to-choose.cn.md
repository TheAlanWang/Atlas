## 核心问题

每个应用都需要存储数据。第一个决策是：**关系型（SQL）还是非关系型（NoSQL）？**

没有放之四海而皆准的答案——正确的选择取决于你的数据结构、访问模式和规模需求。

## 关系型数据库（SQL）

关系型数据库将数据存储在**有行有列的表**中，表之间通过外键定义关联，用 SQL 查询。

```sql
-- 用户表
id | name  | email
1  | Alan  | alan@example.com

-- 订单表
id | user_id | amount
1  | 1       | 99.99
```

**优势：**
- **ACID 保证** — 事务可靠（原子性、一致性、隔离性、持久性）
- **Join 查询** — 高效跨表查询关联数据
- **Schema 约束** — 写入前验证数据结构
- **成熟生态** — 数十年的工具积累、优化经验和专家知识

**常见产品：** PostgreSQL、MySQL、SQLite

**适合场景：**
- 数据有明确关联关系（用户 → 订单 → 支付）
- 需要事务（银行、电商）
- 数据结构稳定、定义清晰

## 非关系型数据库（NoSQL）

NoSQL 数据库抛弃了表/行结构，改用灵活的格式——文档、键值对、图或时序数据。

```json
// MongoDB 文档
{
  "_id": "1",
  "name": "Alan",
  "email": "alan@example.com",
  "orders": [
    { "id": "1", "amount": 99.99 }
  ]
}
```

**优势：**
- **灵活 Schema** — 文档可以有不同的字段
- **水平扩展** — 天生为跨多机器扩展而设计
- **查询性能** — 无需 Join，特定访问模式下读取更快
- **开发友好** — JSON 文档与应用对象天然映射

**常见产品：** MongoDB、Redis、Cassandra、DynamoDB

**适合场景：**
- 数据结构多变或频繁演化
- 需要极高写入吞吐（日志、事件、IoT）
- 数据天然是层次化或嵌套结构

## 对比一览

| | SQL（PostgreSQL）| NoSQL（MongoDB）|
|--|----------------|----------------|
| **数据模型** | 表 + 行 | 文档（JSON）|
| **Schema** | 固定、强制约束 | 灵活 |
| **关联关系** | 跨表 Join | 嵌入或引用 |
| **事务** | 完整 ACID | 有限支持（持续改进）|
| **扩展方式** | 纵向扩展为主 | 横向扩展 |
| **查询语言** | SQL | MongoDB 查询语言 |
| **适合** | 结构化、关联性数据 | 灵活、层次化数据 |

## "直接用 PostgreSQL" 的理由

对大多数应用来说，**从 PostgreSQL 开始**。它久经考验，支持 JSON 列（兼顾 SQL 和 NoSQL），性能出色，能处理大多数场景。

除非有具体的、令人信服的理由，否则不要因为 NoSQL "更时髦"就去用它。

> "PostgreSQL 能做大多数 MongoDB 能做的事，但 MongoDB 做不了大多数 PostgreSQL 能做的事。" — 数据库社区常见说法

## 如何决策

问自己这几个问题：

1. **数据之间有关联关系吗？** → SQL
2. **需要 ACID 事务吗？** → SQL
3. **Schema 不确定或变化很快？** → NoSQL
4. **要大规模存储文档/事件/日志？** → NoSQL
5. **在做普通 Web 应用？** → PostgreSQL（默认首选）

## 关键问题

> _Q: SQL 和 NoSQL 数据库有什么区别？_

SQL 数据库将数据存储在预定义 Schema 的表中，支持 ACID 事务和 Join 查询。NoSQL 数据库使用灵活的数据模型（文档、键值、图），支持水平扩展，适合非结构化或快速变化的数据。SQL 适合关联性、事务性数据，NoSQL 适合灵活的高规模场景。

> _Q: ACID 分别代表什么？_

原子性（Atomicity，事务完全成功或完全失败）、一致性（Consistency，数据始终从一个有效状态转移到另一个有效状态）、隔离性（Isolation，并发事务互不干扰）、持久性（Durability，已提交的事务在系统故障后仍然存在）。ACID 特性是关系型数据库在金融和事务系统中可靠运行的基础。

> _Q: 什么时候选 MongoDB 而不是 PostgreSQL？_

当数据天然是层次化或文档结构、Schema 变化频繁、或需要跨多台服务器水平扩展写入时，选 MongoDB。对大多数有关联数据和事务需求的标准应用，PostgreSQL 是更好的默认选择。
