## 什么是 Hybrid Search

Hybrid search 指的是把向量检索和关键词检索结合起来。

核心思路很直接：

- 向量检索负责语义相似
- 关键词检索负责精确词项匹配

在一些 RAG 系统里，这两种能力结合起来会更有效。

## 为什么单靠向量检索不够

纯向量检索很擅长处理“说法不同但意思接近”的问题。

但当问题依赖精确词项时，它就不一定稳定，比如：

- 产品名
- error code
- API 字段名
- 法务条款
- 版本号

像 `ERR_403_TOKEN_EXPIRED` 这种查询，就不应该只靠语义相似。

## 一个常见流程

```text
query
-> 向量检索
-> 关键词检索
-> 合并分数
-> 得到候选集
```

有些系统会直接混合分数，有些系统会先各自召回，再合并候选集，最后交给 reranker。

## 哪些场景最值得用

当你的语料包含下面这些内容时，hybrid search 往往很有价值：

- 技术文档
- 客服知识库
- 产品规格说明
- 带很多业务术语的文档

因为这些语料同时依赖“语义理解”和“精确词项”。

如果用更面试化的说法，可以直接讲：

`hybrid search 最有价值的前提，是精确词项约束很重要，而纯语义检索在这方面召回不够`

这才是它该被引入的真正原因。

## 你需要理解的代价

Hybrid search 往往能提升召回，但也会增加系统复杂度。

你需要额外考虑：

- 两套检索信号
- 分数归一化
- 结果合并
- 更高的延迟

所以它不是默认必开，而是在关键词信号确实能补足 embedding 短板时才最有价值。

## 什么情况下不值得上

Hybrid search 不是向量检索之后的默认下一步。

下面这些情况下，它往往不值得：

- 语料规模小而且干净
- 精确词项本身不重要
- 纯语义检索已经足够稳定
- 延迟预算很紧

如果真正问题并不在精确词项匹配，上 hybrid search 只会增加复杂度，不会解决核心瓶颈。

## 它和 Reranking 的区别

这两个概念相关，但不是一回事：

- hybrid search 负责把候选集找回来
- reranking 负责把候选集重新排序

很多生产系统会这样组合：

```text
hybrid retrieval
-> merged candidate set
-> reranker
-> final top chunks
```

## 关键问题

> _Q: RAG 里的 hybrid search 是什么？_

Hybrid search 就是把语义检索和关键词检索结合起来。向量部分负责理解语义，关键词部分负责保留对精确词项的匹配能力，比如产品名、错误码或版本字符串。当单一信号不够稳定时，就会用它。

> _Q: 什么情况下 hybrid search 比纯向量检索更有帮助？_

当语料里有很多重要的精确 token，比如产品名、API 参数、工单 id 或法务措辞时，它会更有帮助。更准确地说，是这些场景里精确词项约束很重要，而纯语义检索往往对精确命中的召回不够稳定。

> _Q: Hybrid search 和 reranking 有什么区别？_

Hybrid search 改变的是候选怎么被召回，reranking 改变的是已经召回的候选怎么排序。前者主要提升召回和覆盖，后者主要提升最终相关性。

> _Q: 什么情况下 hybrid search 不值得它带来的复杂度？_

当纯语义检索已经足够好、精确词项并不关键，或者系统更在意延迟和运维简单性时，它往往不值得。它应该是用来解决真实的词项检索缺口，而不是默认配置。
