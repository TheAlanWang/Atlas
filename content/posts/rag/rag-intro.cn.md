## 核心思想

RAG 是 **Retrieval-Augmented Generation（检索增强生成）** 的缩写。名字直接说明了它做什么：在生成回答之前，系统先**检索**相关信息，再用这些信息来**增强**生成过程。

核心逻辑很简单：与其期望一个模型提前知道所有事情，不如在它需要的时候给它提供外部知识。

## RAG 不只属于 LLM

RAG 是一个通用模式，在很多你已经在用的系统里都能看到它的影子：

**搜索引擎**
Google 不是从头生成答案的，它先检索相关网页，再排序呈现。检索这一步就是 RAG。

**推荐系统**
Netflix 不靠纯粹的推理决定推荐什么，它根据你的观看历史检索候选内容，再进行排序。这也是 RAG。

**客服机器人**
一个在回复前先查询你的订单历史的客服机器人，就是在做 RAG。先检索你的数据，再生成相关回复。

**代码助手**
GitHub Copilot 这类工具在给出补全建议之前，会先从大量代码库中检索相似的代码片段。先检索，再生成。

## 通用模式

每一个 RAG 系统都有相同的两步结构：

```
用户输入
   │
   ▼
1. 检索：从外部来源找到相关信息
   │
   ▼
2. 生成：利用检索到的信息产出结果
   │
   ▼
响应
```

不同系统之间的区别在于检索的是什么（网页、文档、数据库记录、代码），以及如何生成（排序、摘要、补全、问答）。

## 为什么 RAG 重要

任何生成系统都有同样的局限：它只能用自己已经知道的东西。RAG 打破了这个限制，把生成与实时的外部知识连接起来。

这让系统更准确、更及时，输出也更有真实依据，而不是基于模糊的近似。

## 本 Topic 的内容

这个 topic 的其余内容专注于 **基于 LLM 的 RAG**：通过检索让大语言模型更准确、减少幻觉。

这是目前 RAG 最广泛的应用形式，也是你在实际工作中遇到的大多数 AI 助手、文档问答工具和知识库聊天机器人背后的技术。

## 面试常问

> _Q: RAG 是什么缩写？它的核心思想是什么？_

RAG 是 Retrieval-Augmented Generation（检索增强生成）的缩写。核心思想是在查询时检索相关的外部信息，用它来增强生成过程，而不是只依赖模型已有的知识。这让输出结果更准确、更及时，也更有真实数据的依据。

> _Q: RAG 是大语言模型专属的技术吗？_

不是。RAG 是一个通用模式，在很多系统中都有体现：搜索引擎在排序结果前先检索网页，推荐系统在打分前先检索候选内容，代码助手在给出补全建议前先检索相似代码片段。基于 LLM 的 RAG 是最主流的现代应用，但"先检索再生成"这个模式早于 LLM 就已经存在了。
